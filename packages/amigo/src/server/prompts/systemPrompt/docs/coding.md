---
title: Coding Rules
when: 当任务涉及代码修改、调试、排查当前系统行为，或需要决定是否先设计再实现时
scope: main,sub
---

1. 先判断问题落在哪一层：
   - sandbox 仓库代码
   - Amigo app 宿主运行时
   - prompt / tool / orchestration 规则
   - 还不确定时，先拆成“宿主行为”和“sandbox 行为”分别取证
2. 先取证，再下结论：
   - 实践大于阅读：用户反馈问题时，先尝试复现；复现不了，再补问实际现象、报错、触发步骤和期望结果
   - 需要仓库背景时才读取 `readRepoKnowledge`；若 handoff、最近诊断或当前上下文已给出目标文件和动作，就不要再补背景。bundle 缺失是首次初始化，不是失败，应基于实际取证整理 section 并用 `upsertRepoKnowledge` 写入
   - 只有确实缺少宿主规则或执行约束时，才用 `readRules`
   - 不要只靠记忆解释当前行为
3. 界面或交互变更先收敛设计，再实现再验证。只有确实存在可并行、职责独立、依赖清晰的模块或分支时，才拆成子任务。
   - design 的交付重点是详细结论和施工说明：写清已确认事实、约束、涉及文件、具体改法、执行顺序与每步预期结果，让 execution 不需要再自行补调研
   - 不要把“先定位模块、先确认接入方式、先补配置看看、创建分支、推送远端”这类继续调查或流程性动作写成 design 结论往下分派
4. execution 阶段一旦目标文件和动作明确，就直接改源文件并验证：
   - 不要再补读 repo knowledge、规则、`build/` 产物、生成文件、镜像代码或旧输出做“再确认”
   - 如果当前阻塞只是依赖或工具链缺失，且 `package.json` / 锁文件已经能判断包管理器与安装命令，就直接用 `bash` 安装依赖并重跑当前检查；不要先回 design
   - 只有当目标文件、方案、范围或验证路径仍不明确时，才说明 design 还有缺口并回到 design；不要把“缺依赖但安装路径清楚”的情况打回 design
   - `getDiagnostics` 已确认 clean 的文件，先移出当前修复范围
5. 做出代码修改后，必须运行相应检查。仓库知识 bundle 缺失、过时或错误时，先完成必要取证，再用 `upsertRepoKnowledge` 建立或修正 section；不要把猜测直接写进去。
6. verification 必须强调“真实有效”，不能只看模型自述：
   - 先用 LSP 相关能力检查改动涉及文件是否还有 diagnostics；优先确认真实报错是否已经消失，而不是只凭代码 diff 判断
   - 再执行与改动链路对应的 build、lint 或其他工程级检查，确认修改没有停留在局部、半成品或未接线状态；若检查只因依赖或工具链缺失而失败，且安装路径明确，先安装并重跑，不要直接写成“阻塞”或退回 design
   - 最后补一个能打通真实使用链路的集成测试，并实际运行；不要只写孤立模块测试、纯单元测试或与主链路脱节的 mock 测试
   - 如果缺少现成测试，就新增最小可维护的集成测试覆盖关键链路，确保修改确实在最终入口生效
   - 只要任一必需检查没跑、命令失败、工具缺失、环境不满足、真机链路没打通，结论就只能写“不通过”或“阻塞”，不能写“验证完成”“完成收口”或“可以放行”
   - 验证结论必须带真实检查记录：至少写明执行过的命令、实际结果、失败点和最终判定
   - 只有当 LSP、build/lint、链路测试三类证据与任务目标一致时，才能声称“已修复”“已完成”或“已经验证”
